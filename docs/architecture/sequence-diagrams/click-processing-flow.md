# Click Processing Flow -- SQS FIFO to Worker Lambda

Asynchronous batch processing of click events from the redirect Lambda.

```mermaid
sequenceDiagram
    participant Redirect as Lambda (redirect)
    participant SQS as SQS FIFO<br/>(shorty-clicks.fifo)
    participant Worker as Lambda (worker)
    participant Clicks as DynamoDB (clicks)
    participant Links as DynamoDB (links)

    Note over Redirect: During redirect, fire-and-forget goroutine
    Redirect-)SQS: SendMessage<br/>MessageGroupId: {code}<br/>MessageDeduplicationId: {uuid}<br/>Body: {code, ip_hash, country,<br/>device_type, referer_domain,<br/>user_agent_hash, timestamp}
    
    Note over SQS: FIFO guarantees:<br/>- Ordering within MessageGroupId (per code)<br/>- Exactly-once delivery (dedup by uuid)<br/>- Batch up to 10 messages per receive

    Note over SQS,Worker: SQS triggers Worker Lambda<br/>Batch size: 10, max batching window: 5s

    SQS->>Worker: Event: Records[] (batch of 1-10 click events)

    Note over Worker: Step 1: Parse and validate batch
    
    loop For each message in batch
        Note over Worker: Deserialize message body<br/>Validate required fields
    end

    Note over Worker: Step 2: Group events by link code
    Note over Worker: {code_A: [event1, event2], code_B: [event3]}

    Note over Worker: Step 3: BatchWriteItem to clicks table
    Worker->>Clicks: BatchWriteItem (up to 25 items per call)<br/>Items: [{PK: LINK#{code}, SK: CLICK#{ts}#{uuid},<br/>ip_hash, country, device_type, referer_domain,<br/>user_agent_hash, created_at (TTL: now+90d)}]

    alt BatchWriteItem partial failure (UnprocessedItems)
        Note over Worker: Exponential backoff retry (up to 3 attempts)
        Worker->>Clicks: BatchWriteItem (unprocessed items only)
    end

    alt Complete failure after retries
        Note over Worker: Return error -> SQS will retry delivery<br/>Failed messages return to queue
        Worker-->>SQS: Error (messages become visible again)
    end

    Clicks-->>Worker: Success

    Note over Worker: Step 4: Increment click_count per code

    loop For each unique code in batch
        Worker->>Links: UpdateItem<br/>Key: {PK: LINK#{code}, SK: META}<br/>Update: SET click_count = click_count + :count<br/>Condition: is_active = :true AND<br/>(attribute_not_exists(max_clicks) OR<br/>click_count < max_clicks)
        
        alt ConditionalCheckFailed (link inactive or limit reached)
            Note over Worker: Log warning, skip increment.<br/>Link is already deactivated or at limit.<br/>Click event is still recorded for auditing.
            Links-->>Worker: ConditionalCheckFailedException
        else Success
            Links-->>Worker: OK (new click_count returned)
            
            alt click_count == max_clicks after increment
                Note over Worker: Link just hit its click limit.<br/>Next redirect will see click_count >= max_clicks<br/>and return 410. No explicit deactivation needed.
            end
        end
    end

    Note over Worker: Step 5: Delete processed messages
    Worker-->>SQS: Success (Lambda runtime auto-deletes on success)
    
    Note right of Worker: Processing time per batch:<br/>10 events: ~50-100ms<br/>DynamoDB BatchWriteItem: ~20ms<br/>UpdateItem per code: ~10ms each
```

## SQS FIFO Configuration

| Property | Value | Justification |
|----------|-------|---------------|
| Queue name | `shorty-clicks.fifo` | FIFO suffix required by AWS |
| MessageGroupId | Short code (e.g., `abc1234`) | Orders clicks per link; different links process in parallel |
| MessageDeduplicationId | UUID (generated by redirect Lambda) | Prevents duplicate click recording on SQS retry |
| Visibility timeout | 60s | 6x the expected processing time (~10s) |
| Receive wait time | 20s | Long polling to reduce empty receives |
| Max receive count | 3 | After 3 failures, move to DLQ |
| Dead-letter queue | `shorty-clicks-dlq.fifo` | Failed events for manual investigation |
| Batch size | 10 | Lambda trigger batch size |
| Max batching window | 5s | Wait up to 5s to fill a batch before invoking |

## Error Handling

| Error | Handling |
|-------|----------|
| Message parse failure | Log error, skip message (it will go to DLQ after max receives) |
| BatchWriteItem partial failure | Retry unprocessed items with exponential backoff (100ms, 200ms, 400ms) |
| BatchWriteItem full failure | Return error from Lambda; SQS retries delivery |
| UpdateItem ConditionalCheckFailed | Expected when link is deactivated; log and continue |
| DynamoDB throttling | AWS SDK automatic retry with exponential backoff |
| Lambda timeout | SQS retries; idempotent writes (same PK/SK) prevent duplicates |

## Idempotency

Click events are inherently idempotent because:
1. Each click has a unique SK (`CLICK#{ts}#{uuid}`). Re-processing writes the same item.
2. SQS FIFO deduplication (by UUID) prevents duplicate delivery within the 5-minute dedup window.
3. `click_count` increments are NOT idempotent -- but the FIFO dedup window prevents double-counting. If a message is somehow processed twice (after the 5-min window), the count will be off by 1, which is acceptable for analytics.
